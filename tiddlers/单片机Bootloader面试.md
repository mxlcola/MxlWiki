## 一、基础概念类（必问，必须背熟）

### 1. 什么是Bootloader？

**标准答案**：

Bootloader是单片机**上电复位后第一个执行的程序**，是芯片启动的引导程序。它不负责业务逻辑，核心工作是**初始化最小硬件、校验应用程序合法性、合法则跳转到App运行，同时支持固件在线升级（IAP/OTA）**。

**原理**：

单片机复位后，PC指针从**Flash起始地址**开始执行，Bootloader就烧录在这个地址，优先级最高。

**面试官深挖**：没有Bootloader会怎样？

答：芯片只能通过J-Link/ST-Link烧录，**无法在线升级、无法程序自检、无法异常恢复**。

---

### 2. Bootloader 和 App（应用程序）的区别？

**标准答案**：

| 维度       | Bootloader              | App 应用程序               |
| ---------- | ----------------------- | -------------------------- |
| 执行顺序   | 上电**第一个运行**      | 被Bootloader引导后运行     |
| 存储地址   | 固定在**Flash起始地址** | 存储在Bootloader后方分区   |
| 功能       | 引导、升级、校验、防护  | 业务逻辑（采集/控制/通信） |
| 中断向量表 | 固定在起始地址          | 必须**重映射**偏移地址     |
| 更新权限   | 一般**不允许随意更新**  | 支持远程/在线升级          |

**核心一句话**：两者**地址完全分离、代码独立、中断表独立**，互不干扰。

---

### 3. Bootloader的核心作用？

1. **上电引导**：初始化时钟/串口/RAM，跳转到合法App

2. **IAP在线升级**：不用仿真器，通过串口/CAN/4G远程更新固件

3. **合法性校验**：防止固件篡改、损坏、跑飞

4. **安全启动**：验签通过才运行，防止恶意程序

5. **异常恢复**：升级失败/程序崩溃，自动回滚备份程序

6. **硬件初始化**：配置最基础的系统时钟、看门狗

---

## 二、核心启动流程（必考，画流程也能答）

### 4. 单片机完整启动流程（Bootloader+App）

**标准答案**：

1. 芯片**上电/复位**

2. 硬件初始化，PC指针指向 **Flash 0地址**

3. 进入**Bootloader**执行

4. 关闭全局中断、初始化最小系统

5. **校验App区固件**（CRC/校验和/签名）

6. 校验合法 → **跳转到App**

7. 校验失败 → 停留Bootloader等待升级，或启动备份固件

8. App初始化中断向量表、外设，执行业务逻辑

**面试官深挖**：为什么要先关中断？

答：跳转前中断还指向Bootloader的地址，不关中断会直接**死机/跑飞**。

## 三、跳转机制（最核心、最容易问原理）

### 5. Bootloader 如何跳转到 App？（STM32通用）

**标准答案**：

跳转必须完成**3个关键步骤**，缺一不可：

1. **关闭所有中断**

2. 关闭全局中断、关闭定时器/串口/DMA中断，防止跳转时触发中断。

3. **设置App的栈指针（MSP）**

4. 从App存储区的**第一个地址**读取栈顶地址，赋值给主堆栈指针。

5. **设置PC指针跳转到App复位向量**

6. 从App存储区**第二个地址**读取复位函数地址，直接跳转。

7. 可选：关闭时钟、关闭看门狗、清理寄存器状态

**代码逻辑（面试口述版）**：

```C
// 1. 关中断
__disable_irq();

// 2. 获取App栈顶指针
uint32_t app_stack = *((__IO uint32_t *)APP_ADDRESS);

// 3. 设置栈指针
__set_MSP(app_stack);

// 4. 获取App复位函数地址
uint32_t app_jump = *((__IO uint32_t *)(APP_ADDRESS + 4));

// 5. 跳转到App
((void (*)(void))app_jump)();
```

**面试官深挖**：跳转失败的常见原因？

答：地址错误、栈设置错误、没关中断、中断向量表没重映射、时钟未复位。

## 四、Flash分区（实战必问）

### 6. Flash 怎么分区？为什么要分区？

**标准答案**：

单片机Flash必须**按功能分区**，防止Bootloader被覆盖、升级变砖。

**标准分区方案（最常用）**：

1. **Bootloader区**：24KB~32KB（存放引导程序）

2. **App A区（主程序）**：用户业务代码

3. **App B区（备份程序）**：升级备用区（防变砖）

4. **参数区**：存放升级标志、版本号、校验值

**一句话总结**：分区是为了**隔离、安全、可回滚**。

---

## 五、IAP / ISP / OTA 区别（高频）

### 7. IAP、ISP、OTA 分别是什么？

**标准答案**：

- **ISP（在线系统编程）**

芯片**出厂自带**的Bootloader，通过串口/USB烧录，只能更新整个程序，**不能自己开发**。

- **IAP（在应用编程）**

**自己编写**的Bootloader，程序运行中更新固件，支持断点续传、校验、回滚。

- **OTA（空中升级）**

通过4G/NB-IoT/WiFi远程下发升级包，本质是**IAP+无线通信**，属于远程IAP。

**最简记忆**：

ISP=厂家自带；IAP=自己写的本地升级；OTA=远程无线升级。

## 六、中断向量表（必踩坑、必问）

### 8. App偏移地址后，为什么程序不跑？怎么解决？

**标准答案**：

因为**中断向量表默认在Flash 0地址**（Bootloader位置），App偏移后，中断找不到入口，直接死机。

**解决方法**：

在App初始化时，**重映射中断向量表偏移地址**：

```C
SCB->VTOR = FLASH_BASE | APP_OFFSET_ADDRESS;
```

**面试官深挖**：Bootloader需要重映射吗？

答：不需要，Bootloader在0地址，默认向量表就是正确的。

## 七、固件升级流程（项目必问）

### 9. 完整IAP/OTA升级流程？

**标准答案**：

1. 上位机/云端下发**升级包**（bin文件）

2. 单片机组包、校验（CRC32）

3. 写入**App B备份区**（不覆盖主程序）

4. 写入完成后整体校验合法性

5. 设置**升级标志**

6. 重启 → Bootloader检测到升级标志

7. 把B区程序复制到A区（或直接跳转B区）

8. 启动成功 → 清除升级标志

9. 启动失败 → 自动回滚旧版本

---

## 八、安全与防变砖（大厂必考）

### 10. 如何保证升级不断电、不变砖？

**标准答案（5条安全机制）**：

1. **双分区备份（A/B区）**

2. 升级写备用区，不破坏主程序，失败不影响运行。

3. **升级状态标记**

4. 用Flash参数区记录：升级中/升级成功/升级失败。

5. **整体校验**

6. 整包CRC/SHA256校验，不合法不启动。

7. **断电续传**

8. 记录当前包号，上电继续传输。

9. **自动回滚**

10. 启动超时/崩溃，自动切换回旧版本。

---

### 11. 如何防止固件被篡改/破解？

1. **固件加密**：AES-128加密升级包

2. **签名验签**：SHA256+RSA/ECC，Bootloader验签通过才运行

3. **校验值保护**：CRC32校验固件完整性

4. **加密传输**：升级包加密下发，防止窃听

---

## 九、开发常见坑点（加分项）

### 12. Bootloader开发最容易踩的8个坑

1. **链接脚本地址写错** → 跳转直接死机

2. **跳转前没关中断** → 跑飞

3. **App中断向量表未重映射** → 不进中断、死机

4. **栈空间设置错误** → 堆栈溢出

5. **时钟未复位** → App时钟错乱

6. **升级写主程序区** → 断电直接变砖

7. **未做校验** → 损坏固件也运行

8. **看门狗未喂狗/未关闭** → 跳转时复位

---

## 十、拔高问题（大厂/高端岗）

### 13. 什么是差分升级？

**答**：只传输**新旧固件差异部分**，不用传完整bin包，大幅节省流量/时间，适合低带宽场景（NB-IoT、LoRa）。

### 14. 什么是安全启动（Secure Boot）？

**答**：链式验签机制

1. Bootloader（芯片信任根）

2. 验App签名

3. 验通过 → 运行

4. 验不通过 → 拒绝启动

防止恶意固件、固件篡改。
